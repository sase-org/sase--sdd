# Improve `/sase_changespecs` Current CL/PR Detection

## Goal

Make the `/sase_changespecs` skill tell agents how to identify the ChangeSpec associated with the current CL or PR, even
when the user did not provide the ChangeSpec name explicitly.

## Context

The live provider skill files are generated. The source of truth is `src/sase/xprompts/skills/sase_changespecs.md`;
generated `SKILL.md` files should not be edited directly. After changing the source, regenerate with
`sase init-skills --force` and deploy with the existing chezmoi flow.

Relevant existing behavior:

- `sase search '<query>' -f markdown` searches across active and archive `.gp` files.
- Exact ChangeSpec name lookup is `sase search '&<changespec_name>' -f markdown`.
- Free-text search includes the ChangeSpec `CL` / `PR` field, so quoted PR URLs, CL URLs, PR numbers embedded in URLs,
  and branch-like ChangeSpec names are usable lookup keys.
- Agent/commit workflows already prefer `SASE_AGENT_CL_NAME` when available, then fall back to the current VCS branch or
  bookmark through `get_cl_name_from_branch()`.
- Git/GitHub branches can diverge from ChangeSpec names because of suffix handling and branch aliases, so the skill
  should present branch lookup as a strong clue but require confirmation with `sase search`.

## Proposed Skill Guidance

Add a new workflow section near the top of `/sase_changespecs`, before generic blocking/status workflows:

1. Check for explicit SASE context first:
   - `printf '%s\n' "$SASE_AGENT_CL_NAME"`
   - If it is non-empty, run `sase search '&<value>' -f markdown`.
   - If that misses, run `sase search '"<value>"' -f markdown` because the value may be a branch, alias, or text
     recorded elsewhere.

2. Check the current VCS branch/bookmark:
   - For git repos: `git branch --show-current`.
   - Search it exactly as a ChangeSpec name first, then as free text: `sase search '&<branch>' -f markdown`, then
     `sase search '"<branch>"' -f markdown`.
   - Confirm the match by comparing project, `PR`/`CL`, and the file location in the result.

3. If the current PR/CL URL is available, search by the URL:
   - GitHub: `gh pr view --json url,number,headRefName,baseRefName`.
   - Search the URL with `sase search '"<url>"' -f markdown`.
   - If only a number is available, search the URL fragment or number as quoted text, then disambiguate by project and
     branch.
   - For provider-specific CL tools, get the current change URL/number if available and search it the same way.

4. If multiple candidates match:
   - Prefer the candidate whose `PR`/`CL` field matches the current URL.
   - Then prefer the one whose `NAME` matches `SASE_AGENT_CL_NAME` or the current branch.
   - Then prefer the one in the project inferred from the repository (`get_workspace_name` equivalent / repo name).
   - Report ambiguity instead of guessing silently.

5. If no candidate matches:
   - Say no associated ChangeSpec was found.
   - Include the signals checked: env value, branch/bookmark, PR/CL URL or number, and project.
   - Do not fabricate a ChangeSpec.

## Risks and Mitigations

- Risk: agents may treat branch name as definitive when GitHub branch aliases or suffix stripping apply. Mitigation:
  instruct the skill to confirm with `sase search` output and prefer exact PR/CL URL matches.
- Risk: numeric PR/CL searches may produce unrelated matches. Mitigation: require quoted searches and project/branch
  disambiguation, and tell agents to report ambiguity.
- Risk: directly editing generated skill files would be overwritten. Mitigation: edit only
  `src/sase/xprompts/skills/sase_changespecs.md` and regenerate.
