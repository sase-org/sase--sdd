# Plan: Add a Context-Aware Command Palette to the Ace TUI

## Goal

Add a beautiful, reliable command palette to `sase ace` that opens when the user presses `:`. The palette should show
only commands that are applicable to the current tab and selected entry, while still giving every existing TUI keymap a
representation somewhere in the command system.

The intended UX is:

- Press `:` from the normal TUI to open a modal command palette.
- Type to fuzzy-filter commands by label, key, action name, tab, or short aliases.
- See the active keybinding beside each command, including multi-key forms such as `%n`, `,t`, or `zc`.
- Press Enter to run the highlighted command and close the palette.
- Press Esc to cancel without changing app state.
- The list adapts to the focused tab (`changespecs`, `agents`, `axe`) and the selected row on that tab.

This should not duplicate the help modal. Help remains the full reference. The command palette is an action surface: it
should show runnable commands for the user’s current context, with disabled/inapplicable commands omitted from the
default list.

## Current State

- App-level keymaps are centralized in:
  - `src/sase/ace/tui/keymaps/types.py`
  - `src/sase/ace/tui/keymaps/loader.py`
  - `src/sase/default_config.yml`
- Runtime Textual bindings are generated by `build_app_bindings()` from `_BINDING_META`, plus hardcoded digit bindings.
- Prefix modes exist for:
  - fold mode (`z...`)
  - copy mode (`%...`)
  - leader mode (`,...`)
  - bang mode (`!...`)
  - user-defined custom modes
- Key dispatch is still spread across app action methods and mode handlers:
  - normal actions use `action_*` methods on `AceApp`
  - prefix modes use `_handle_fold_key`, `_handle_copy_key`, `_handle_leader_key`, `_handle_bang_key`,
    `_handle_custom_mode_key`
  - some actions change meaning by tab, for example `run_workflow`, `show_diff`, `edit_spec`, `open_tmux`, and
    `kill_agent`
- The footer already computes some entry-dependent availability in `src/sase/ace/tui/widgets/_keybinding_bindings.py`,
  but it intentionally only displays conditional bindings.
- The help modal in `src/sase/ace/tui/modals/help_modal/` contains broad tab-specific keybinding reference sections, but
  it is not an executable command registry.
- `AceApp.ENABLE_COMMAND_PALETTE = False`, so Textual's built-in command palette is intentionally disabled. This feature
  should use a SASE-specific modal rather than enabling Textual's generic palette.

## Design Principles

1. **Single command catalog** Build a real command catalog instead of scraping the help modal or footer. Help/footer can
   later consume it, but the palette should not depend on their presentation code.

2. **Every keymap represented** Every `AppKeymaps` field, every built-in mode key, every saved-query digit binding, and
   every custom mode command should have a `CommandSpec`. Multi-key mode commands should be represented directly, not
   hidden behind a generic "enter copy mode" command.

3. **Applicability is explicit** Each command should declare a scope and/or predicate over a small context object:
   - current tab
   - selected ChangeSpec, if any
   - selected Agent, if any
   - selected AXE item, if any
   - mark counts, runner count, completed-agent count, focused group/banner state, file-panel state, attempt state
   - app mode state where relevant

4. **Palette execution reuses existing behavior** Selecting a command should call existing `action_*` methods or
   existing mode handlers. Do not reimplement business behavior inside the palette.

5. **No inapplicable commands by default** The normal palette view should omit unavailable commands, per the user
   request. Tests should lock this in for representative CL, Agent, and AXE states.

6. **Beautiful terminal UI** Use the existing visual language: centered modal, double border, `#00D7AF` for key accents,
   `#87D7FF` for section/tab accents, `#FFD700` for active/highlight accents, and concise footer hints. The layout
   should stay legible on narrow terminals and not shove long labels into awkward wrapping.

## Proposed Architecture

Add a new command subsystem under `src/sase/ace/tui/commands/`:

- `types.py`
  - `CommandContext`
  - `CommandSpec`
  - `CommandAvailability`
  - `CommandPaletteResult`
  - tab/category enums or literals
- `catalog.py`
  - static catalog builders for app, digit, built-in mode, and custom mode commands
  - command metadata derived from `KeymapRegistry`
- `context.py`
  - one small adapter that extracts the current command context from `AceApp`
- `availability.py`
  - pure predicates that decide which commands are applicable
  - should share helper logic with footer availability where possible
- `execute.py`
  - dispatch selected commands back to app actions or existing mode handlers

Add a new modal:

- `src/sase/ace/tui/modals/command_palette_modal.py`
  - `Input` at the top for filtering
  - scrollable command list
  - highlighted row with key, label, category/tab badge, and optional short context detail
  - Enter executes, Esc cancels, Up/Down or Ctrl+P/Ctrl+N navigates

Add one new app action:

- `action_open_command_palette`
  - bound to `colon`
  - pushes the palette with the current context and registry
  - on result, dispatches through `commands.execute`

Update key validation/display:

- add `colon` to `_KEY_DISPLAY`
- add `open_command_palette` to `AppKeymaps`, `_BINDING_META`, and `default_config.yml`

## Phase 1: Command Catalog and Applicability Core

Owner: first implementation agent.

Scope:

- Create the `src/sase/ace/tui/commands/` package.
- Define `CommandContext` and `CommandSpec`.
- Build a command catalog from `KeymapRegistry`.
- Add `open_command_palette: "colon"` to the app keymap schema and defaults.
- Add `colon` support to key display/validation.
- Add focused unit tests for catalog coverage.

Important details:

- The catalog must cover:
  - all fields in `AppKeymaps`
  - saved-query digit bindings `0` through `9`
  - every built-in mode subcommand for fold/copy/leader/bang
  - every valid user-defined custom mode command
- A `CommandSpec` should include:
  - stable id, for example `app.refresh`, `copy.agents.name`, `leader.task_queue`, `saved_query.3`
  - display label
  - key sequence as Textual keys and as display text
  - category, such as `Navigation`, `CL Actions`, `Agent Actions`, `Copy`, `Leader`, `Axe`
  - tabs where the command can ever appear
  - executor descriptor, for example app action name, mode handler name/key, saved-query action, or custom mode command
  - search aliases
- Applicability should be implemented as pure functions over `CommandContext`. Start with parity for the existing footer
  conditional logic and the help modal's tab scoping, then refine entry-specific predicates.
- Avoid circular imports: command modules should import models/types where needed, but should not import `AceApp` except
  under `TYPE_CHECKING`.

Acceptance:

- `test_keymaps.py` updated for the extra app binding count and default-config coverage.
- New command catalog tests prove every configured keymap has a command representation.
- New applicability tests cover representative unavailable cases:
  - no CL selected
  - submitted/reverted CL
  - CL without `cl`
  - agent group banner focused
  - done/failed/running/waiting-input agent
  - AXE parent row versus background command row
  - done AXE command row with re-run available

## Phase 2: Palette Modal UX

Owner: second implementation agent.

Scope:

- Implement `CommandPaletteModal`.
- Add TCSS styling for the modal.
- Add modal-level tests for filtering, highlight movement, cancellation, and selection.

UX requirements:

- The modal should occupy roughly 70-80% width, capped for very wide terminals, and roughly 60-75% height.
- Top row:
  - title: `Command Palette`
  - subtle context badge: `CLs`, `Agents`, or `AXE`
- Input placeholder should be short, for example `Type a command...`.
- Rows should show:
  - key sequence in accent color
  - command label in primary text
  - compact category/tab badge in muted color
  - optional context suffix where it improves clarity, for example `3 marked`, `done command`, `running agent`
- Empty state should be quiet and useful: `No matching commands`.
- Footer should be concise: `Enter run · Esc close · ↑/↓ move`.

Filtering behavior:

- Case-insensitive substring matching is required.
- Simple token matching is enough; fuzzy ranking can be light and deterministic.
- Prefer exact prefix matches, then label matches, then alias/action/category matches.
- Searching by key should work for both display text and raw textual key names, e.g. `ctrl d`, `Ctrl+D`, `leader`, `%n`.

Acceptance:

- Modal renders with a nonempty applicable command list.
- Filtering narrows the list without losing current highlight bounds.
- Enter returns the highlighted command id.
- Esc returns `None`.
- Rows are stable and deterministic for tests.

## Phase 3: App Wiring and Command Execution

Owner: third implementation agent.

Scope:

- Add `action_open_command_palette` to the TUI.
- Bind `:` through config-driven keymaps.
- Extract live `CommandContext` from `AceApp`.
- Execute selected commands by delegating to existing actions/mode handlers.
- Add integration tests using the existing Textual pilot patterns.

Execution strategy:

- App-level commands call `app.action_<name>()`, matching Textual action names.
- Saved-query commands call the existing `action_load_saved_query_N` methods.
- Built-in mode commands should avoid forcing users through transient prefix mode. Dispatch directly by calling the
  corresponding existing handler with the subkey where that is reliable:
  - fold: `_handle_fold_key(subkey)`
  - copy: `_handle_copy_key(subkey)` after setting copy mode state if needed, or better extract copy dispatch helpers so
    palette execution does not depend on footer-mode state
  - leader: `_handle_leader_key(subkey)`
  - bang: `_handle_bang_key(subkey)`
- Custom mode commands should reuse `_handle_custom_mode_key(subkey)` or extract a small helper that runs action/shell
  specs without requiring transient mode state.
- Command execution must close the palette before running actions that suspend, open editors, or push other modals.

Context extraction notes:

- For CLs:
  - selected `ChangeSpec` is `changespecs[current_idx]` when present
  - include mark count
  - include hidden/filter state where relevant
- For Agents:
  - selected agent may be `None` when a collapsed group banner is focused
  - include `_current_group_key`, marked count, completed count, attempt pinned/current attempt, file visibility, and
    whether a CL jump target resolves
- For AXE:
  - selected item comes from `_axe_items[current_idx]`
  - include whether it is the parent, a lumberjack, or a bgcmd
  - include whether a selected bgcmd is done/runnable

Acceptance:

- Pressing `:` opens the palette from each tab.
- Commands visible from the palette match the current tab and selected entry state.
- Selecting representative commands triggers the same behavior as the keybinding:
  - refresh
  - edit query
  - jump to entry
  - CL diff only when a CL exists
  - agent kill/dismiss labels and execution for running/done/group/marked states
  - AXE start/stop/kill and re-run conditions
  - copy-mode command such as copy agent name
  - leader command such as task queue
- `:` does not interfere with prompt text areas, modal inputs, or existing input widgets.

## Phase 4: Unify Help/Footer Metadata and Polish

Owner: fourth implementation agent.

Scope:

- Reduce drift between the command catalog, footer, and help modal.
- Improve command labels/categories based on real screenshots or test render snapshots.
- Add guard tests so future keymaps cannot be added without palette representation.

Recommended approach:

- Keep the footer's compact conditional display, but source labels/keys from the command catalog where practical.
- Keep the help modal's rich two-column reference, but use command catalog metadata for key sequences and labels where
  it does not make the layout worse.
- Add a regression test that fails if:
  - `AppKeymaps` has a field not represented by the command catalog
  - a built-in mode key is missing from the command catalog
  - a command claims applicability for no tab at all

Visual polish:

- Add category ordering that mirrors current help sections rather than sorting everything alphabetically.
- Ensure long labels truncate cleanly in the row, not in the key column.
- Keep selected-row styling high contrast but consistent with existing `OptionList` highlights.
- Confirm there is no single-hue palette drift; use the existing teal/blue/gold accents sparingly.

Acceptance:

- Help, footer, and palette use consistent key display names after custom keymap overrides.
- Adding a new keymap requires adding or generating a command representation, enforced by tests.
- Visual layout remains readable in narrow and wide terminal sizes.

## Open Design Decisions

- Whether inapplicable commands should ever be discoverable behind an alternate filter such as `!all`. The user request
  says users should only see applicable commands, so this plan keeps the default strict and does not add an all-command
  mode unless requested later.
- Whether palette execution of prefix-mode commands should call existing `_handle_*_key` methods directly or first
  extract dedicated pure dispatch helpers. Direct calls are faster to implement; extracting helpers is more robust for
  copy/custom modes whose handlers assume transient mode flags.
- Whether help/footer unification should happen immediately or after the palette lands. This plan places it in Phase 4
  so the first three phases can ship a working palette without destabilizing established help/footer behavior.

## Risks

- **Applicability drift:** The existing action methods are the final source of truth and often silently no-op. The new
  predicates need tests or the palette will show stale/inaccurate commands.
- **Mode-handler assumptions:** Some `_handle_*_key` methods may assume mode flags are active and may refresh footers.
  Phase 3 should extract small dispatch helpers where direct calls create weird UI state.
- **Key collisions:** `:` must be added to validation/display and default config. If a user remaps it, the palette
  should follow their configured key like every other command.
- **Terminal layout:** A command palette can easily become noisy. Keep labels concise, put context in muted suffixes,
  and avoid stuffing full help text into rows.
