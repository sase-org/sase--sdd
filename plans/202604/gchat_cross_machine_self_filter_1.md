# retired chat plugin - Cross-Machine Self-Message Filter

## Problem

The current self-message fix is host-local. `gchat_client.send_message()` records returned Chat message IDs in
`~/.sase/gchat/self_message_ids.json`, and inbound drops messages whose IDs appear there. That works only when the same
machine both sends and polls.

When another machine starts `sase_chop_gc_inbound` against the same space, it has no record of messages sent by the
first machine. Since the `gchat` CLI authenticates as the user's Google identity, those messages still appear as
`sender.type == "HUMAN"`, so the second machine can treat prior sase outbound notifications or launch confirmations as
new user prompts and launch agents.

There is a second state bug that makes this easier to trigger: `last_seen_create_time` is not scoped by
`SASE_GCHAT_SPACE_ID`. If a machine has a stale high-water mark from an old space, then switching to a new empty space
does not look like a first run. Messages in the new space with `createTime > last_seen` are dispatched instead of being
used to seed the new-space cursor.

The user's additional context says the Google Chat space has been changed to a new empty one, so we do not need to
support filtering old messages that were sent before this new fix ships. We do need the next fix to be correct from now
on, including with multiple machines.

## Root Cause

1. The previous sent-ID allowlist is local disk state and is not shared across machines.
2. Google Chat message sender metadata cannot distinguish "typed by me" from "posted by sase using my credentials".
3. Inbound cursor state is global across all configured spaces, so changing spaces reuses unrelated high-water marks.
4. Pure ID recording is therefore insufficient: another host can only classify a self-message if the message itself
   carries a durable signal, or if all hosts share an external store. This plugin has no shared store, so the durable
   signal must live in Chat.

## Goals

1. A message posted by sase on machine A must be ignored by inbound on machine B.
2. Real user messages typed into Chat must still launch agents or satisfy replies as before.
3. Switching `SASE_GCHAT_SPACE_ID` must not reuse another space's inbound cursor.
4. Keep the existing sent-ID store as a local fast path and defense-in-depth.
5. Keep the change centralized so future send call sites inherit the protection.

## Non-Goals

- No attempt to retroactively classify old unmarked messages. The user moved to a new empty space, so pre-fix backlog
  compatibility is not required.
- No shared database, Google Drive state, or cross-host filesystem dependency.
- No removal of the current local self-message ID cache.
- No change to the basic inbound feature set: free-form text still launches agents.

## Design

### 1. Add a durable outbound marker to messages created by `gchat_client`

Introduce a small module, likely `retired_chat_plugin/message_markers.py`, responsible for:

- `mark_outbound_text(text: str) -> str`
- `strip_outbound_marker(text: str) -> str`
- `has_outbound_marker(text: str) -> bool`

The marker must be appended centrally in `gchat_client.send_message()` and in `gchat_client.upload_file()` captions, not
sprinkled through formatting code. That makes the contract apply to workflow notifications, launch confirmations,
dot-command replies, response confirmations, two-step prompts, and attachment uploads.

Marker choice:

- Use an ASCII sentinel line first, for reliability through the `gchat` CLI and `list_messages`, e.g.
  `<!-- retired chat plugin:outbound:v1 -->`.
- Keep marker handling isolated so we can swap the representation later if live Chat rendering proves too visible.
- Make `mark_outbound_text()` idempotent so retries or nested calls do not duplicate the marker.

The existing local ID allowlist remains useful, but it can no longer be the only signal.

### 2. Filter marked messages before dispatch

Update inbound message text handling:

- `_filter_and_sort()` should drop any message where `_msg_text(message)` has the outbound marker, in addition to the
  existing `sender.type == "BOT"` and local recorded-ID checks.
- `_msg_text()` should strip the outbound marker before returning text for normal dispatch paths. This prevents a marker
  from leaking into prompts if a marked message reaches a fallback path due to a future bug.

Filtering in `_filter_and_sort()` is still the key behavior, because filtered self-messages should not advance the
high-water mark by themselves.

### 3. Mark file uploads too

`upload_file()` currently allows `text=None`. Attachment-only outbound uploads from sase can appear as Chat messages and
would be interpreted as inbound photo prompts on another machine.

Update `gchat_client.upload_file()` so:

- If `text` is provided, append the outbound marker to that caption.
- If `text` is missing, send a marker-only caption.

This may make a marker visible on attachment uploads if Chat does not hide the chosen sentinel. That is still preferable
to cross-machine self-triggering, and the marker implementation is isolated for later UX adjustment.

### 4. Scope inbound high-water mark by space

Replace the single `~/.sase/gchat/last_seen_create_time` value with per-space state while preserving compatibility:

- Add helpers in `retired_chat_plugin.inbound`, e.g. `_space_state_key(space_id)` and
  `get_last_seen_create_time(space_id: str | None = None)`.
- Store per-space cursors in a JSON file such as `~/.sase/gchat/last_seen_create_time_by_space.json`.
- Keep reading the legacy flat file only as a fallback when no space is supplied, so existing unit tests and any
  internal callers remain easy to migrate.
- In `sase_gc_inbound.main()`, call `get_space_id()` before reading the cursor and use the space-scoped helpers.

With this change, moving to a new empty Chat space reliably triggers the bootstrap branch on the first inbound run for
that space. That covers any unmarked messages already present in the new space at rollout time.

### 5. Keep state and docs honest

Update docs to describe:

- The local ID cache is only an optimization / same-host guard.
- Cross-machine self filtering is marker-based.
- Inbound high-water state is per Chat space.
- Legacy `last_seen_create_time` may remain on disk but is no longer the main state for `main()`.

## Tests

Add focused unit/integration coverage:

1. `message_markers` tests:
   - marker is added once;
   - marker is detected;
   - marker is stripped cleanly from marked text;
   - empty text can be marker-only for upload captions.
2. `gchat_client.send_message` test:
   - `_run` receives marked text;
   - returned message ID is still recorded in `self_messages`.
3. `gchat_client.upload_file` tests:
   - provided caption is marked;
   - missing caption becomes marker-only text;
   - returned message ID is still recorded.
4. Inbound filter tests:
   - a `HUMAN` message with the marker is dropped even when not in local `self_message_ids.json`;
   - a real unmarked `HUMAN` message is kept;
   - a marked attachment message is dropped and does not call `_handle_photo` / `_launch_agent`.
5. Space-scoped cursor tests:
   - saving and loading for one space does not affect another;
   - `main()` bootstraps a new space even if a legacy/global cursor exists;
   - an existing cursor for the same space still dispatches fresh user messages.
6. Regression integration:
   - simulate machine A by returning a marked outbound message from `list_messages`;
   - simulate machine B with no local self-message ID entry and an old legacy cursor;
   - inbound must not launch an agent.

## Risks and Tradeoffs

- If Google Chat renders the ASCII sentinel visibly, users may see a small marker line. This is an acceptable first
  tradeoff because correctness matters more than cosmetics, and the marker representation is isolated for replacement.
- If the `gchat` CLI strips HTML comments before sending or listing, the marker test with mocked CLI will pass but live
  filtering would fail. The manual smoke test must verify the marker survives the real CLI. If it does not, switch the
  marker constant to a plainly visible bracketed token, e.g. `[retired chat plugin:outbound:v1]`, in the isolated marker module.
- Per-space cursor migration changes state layout. Keeping legacy helpers compatible reduces test churn and avoids
  surprising external callers, while `main()` gets the safer space-scoped behavior.
