## Known measurement caveats

These are not new findings — they are the caveats Phase 7B/7C/7D/7E already recorded, restated here so a Phase 8 reader
does not have to re-derive them.

- **`evaluate_query_many` regresses on the routed path** (`0.05×–0.08×` vs the optimised Python batch path) because
  per-call wire conversion dominates at small-batch sizes. This is the headline Phase 8 sequencing constraint: the
  Python implementation must remain dispatchable for `evaluate_query_many` until either the wire conversion is
  amortised inside Rust or the caller batches differently. Phase 7E intentionally excluded this surface from the floor.
- **Status / Git micro-normalizers (`read_status_from_lines`, `plan_status_transition`,
  `parse_git_branch_name`, `derive_git_workspace_name`, `parse_git_conflicted_files`, `parse_git_local_changes`,
  `parse_git_name_status_z`)** all sit at sub-50 µs medians where dispatch overhead exceeds the win. Phase 7E excluded
  them from the floor; Phase 8 should plan to keep them on the dispatch path or re-evaluate the boundary cost rather
  than treating them as established Rust wins.
- **`sase ace` cold open is 0.84× on the Pilot harness** that mocks the Rust hot paths. The real-user signal
  (`sase agents status -j` cold subprocess at 2.59× synthetic / 2.03× home tree) is the headline for end-to-end value.
- **Floor tolerance is 1.40×, not the plan's recommended 25–35%.** Phase 7E documented the rationale: GitHub-hosted
  runners are noisier than the Phase 7B capture workstation. Phase 7F endorses keeping `1.40×` for the first weeks of
  CI history; the relative `must_beat_python` check is the actual hardware-independent regression detector and covers
  4 of the 5 anchors. Re-evaluate after `phase7-perf-floor` accumulates a few weeks of green runs and ratchet down
  toward `1.25–1.30×` if the runners stay quiet.
- **End-to-end home-tree captures are gitignored** (`plans/202604/perf_artifacts/local_only/home_tree_*.json`) because
  they reflect a workstation-specific tree. The synthetic captures are committed.
- **Stale `.cpython-314.so` shadows the abi3 wheel in long-lived workspaces.** During Phase 7F verification a stale
  `sase_core_rs.cpython-314-x86_64-linux-gnu.so` from an earlier `maturin develop` left the new abi3 wheel masked,
  exposing only the Phase 1–3 bindings (the Phase 4/5 status + Git helpers were absent). Removing the stale .so and
  re-running `just install` resolved it. Worth keeping in mind when reproducing these numbers from a workspace that has
  not been re-installed since the Phase 4/5 sase-core merges; a `.venv`-from-scratch build avoids the issue. This is a
  workspace-hygiene caveat for reproduction, not a defect in the shipped wheel.
