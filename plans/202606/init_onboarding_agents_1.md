# Fix `sase init` AGENTS onboarding

## Diagnosis

Running `sase init` from a temporary copy of `~/projects/github/bbugyi200/bob-cli` reproduces the issue:

```text
Blockers:
  init memory: /tmp/sase-init-repro-bob-cli: unreferenced memory file memory/cli_rules.md
```

The root cause is an interaction between two existing init contracts:

- `sase memory init` validates that every non-README memory note is reachable from `AGENTS.md`. `bob-cli` has
  `memory/cli_rules.md` as a long note with `parent: AGENTS.md`, but its current `AGENTS.md` only contains
  `@memory/sase.md`, so validation blocks.
- The rich AGENTS template used in this repo is generated by the AMD init path only when `amd_h1_title` is configured.
  `bob-cli` has no `amd_h1_title` in `sase.yml`, so bare `sase init` treats the existing minimal `AGENTS.md` as
  acceptable and never rewrites it to include the Tier 1 / Tier 2 sections.

That means the command can detect the broken memory graph but cannot repair it in this repo shape, even though the
repair is deterministic: generate managed AGENTS content that references short memory and lists top-level long memory.

## Proposed behavior

Make bare `sase init` capable of bringing a partially initialized repo like `bob-cli` to the same managed-instructions
shape as this repo:

- If a project has SASE memory files but lacks an AMD title, derive a sensible default project title for onboarding
  instead of leaving AMD inactive.
- Use that derived title in AMD planning and memory synchronization so `AGENTS.md` can be created or overwritten with
  the managed template that includes all top-level long memory notes.
- Preserve explicit configuration precedence: an `amd_h1_title` in project config still wins, invalid configured values
  still block, and home / chezmoi behavior should keep using the existing user-config title rules.
- Keep explicit `sase amd init` conservative if needed, but update bare `sase init` and `sase memory init` paths enough
  that the onboarding flow can repair the broken memory graph.

The default title should be stable and local, probably based on the project directory name, for example:

```text
<project-name> - Agent Instructions
```

For `bob-cli`, this would produce `# bob-cli - Agent Instructions` and the expected Tier 1 / Tier 2 AGENTS structure.

## Implementation plan

1. Add a small AMD title resolution mode for onboarding / memory sync:
   - Keep the existing explicit config loader.
   - Add an opt-in fallback title for project roots when SASE memory exists and no `amd_h1_title` is configured.
   - Avoid applying that fallback to home roots unless the existing user config path already supplies a title.

2. Thread that mode through the existing planners:
   - `plan_amd_init_for_check()` should use the onboarding fallback for bare `sase init`.
   - `build_amd_init_plan()` / `_build_single_amd_init_plan()` should accept the fallback mode instead of inferring all
     behavior from `explicit`.
   - `plan_amd_memory_sync()` should accept the same mode so `sase memory init` can plan/write managed `AGENTS.md`
     before validation.

3. Update memory init planning/application:
   - When `enable_amd=True` for the project root, call AMD memory sync with onboarding fallback enabled.
   - Ensure the overlay used for unreferenced-memory validation includes the rendered managed AGENTS content so long
     memory is not incorrectly blocked before the fix is written.

4. Add regression tests:
   - A `bob-cli`-shaped project with `sase.yml` containing only `sdd.version_controlled`, `memory/sase.md`,
     `memory/cli_rules.md`, and minimal `AGENTS.md` should plan an `AGENTS.md` overwrite and report no memory blocker.
   - Bare `sase init --yes` for that shape should run AMD/memory/SDD as needed rather than stopping on the unreferenced
     long memory blocker.
   - Existing behavior with explicit `amd_h1_title` should remain unchanged.
   - Existing invalid `amd_h1_title` should still block.

5. Verify with focused tests first:
   - `pytest tests/main/test_amd_init.py tests/main/test_init_memory_plan.py tests/main/test_init_onboarding_flow.py`
   - Add any new test file to that focused set if needed.

6. Run repository validation after code changes:
   - `just install` if the workspace environment needs refreshing.
   - `just check` as required by this repo’s instructions.

7. Validate against the real scenario without mutating `bob-cli` first:
   - Re-run `sase init --check` in a fresh temporary copy of `bob-cli`.
   - Then run `sase init --yes` in that copy and inspect generated `AGENTS.md`.
   - Only after the code is verified, report whether running the fixed command in the real `bob-cli` repo is now safe.
