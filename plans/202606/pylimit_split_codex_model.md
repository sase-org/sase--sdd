# Plan: Route pylimit split agents through #m_codex

## Context

The project-local `sase/pylimit_split` xprompt workflow is implemented in `xprompts/pylimit_split.yml`. It is a single
hidden Python step that scans `src/` and `tests/` for Python files over the configured line-count thresholds, builds one
prompt per offending file, joins those prompts into a wait-chained multi-prompt, and launches that multi-prompt through
`sase.agent.launcher.launch_agent_from_cwd()`.

Each generated split-agent segment currently has this shape:

```text
%name:<candidate>
#gh:sase %group:chop %approve #split_file:<path>
```

The requested behavior is to make those generated split agents explicitly target `codex/gpt-5.5` by adding the existing
`#m_codex` xprompt to their prompts. In this environment, `sase xprompt explain m_codex` shows that `#m_codex` expands
to `%model:#codex`, and `sase xprompt explain codex` shows that `#codex` expands to `gpt-5.5`.

The relevant runtime path expands xprompts before extracting directives in `src/sase/axe/run_agent_directives.py`, so a
launched prompt containing `#m_codex` should produce the intended model directive for agent metadata and provider/model
resolution.

## Scope

This change should target the per-file split agents generated by `xprompts/pylimit_split.yml`. It should not change the
outer `sase_pylimit_split` chop configuration or the public `#split_file` prompt. The outer workflow launch is owned by
chop/user configuration, while the per-file child prompts are the workflow-owned agent prompts that need the explicit
model targeting.

No workflow schema update is needed because the workflow step structure and fields are unchanged. No SASE default-config
model alias change is planned because the user explicitly asked to use the existing `#m_codex` xprompt rather than
inlining `%model:codex/gpt-5.5`.

## Implementation

1. Update `xprompts/pylimit_split.yml` so each `per_file_prompts.append(...)` generated segment includes `#m_codex`.

   Preferred shape:

   ```text
   %name:<candidate>
   #m_codex
   #gh:sase %group:chop %approve #split_file:<path>
   ```

   Keeping `#m_codex` on its own line makes the generated prompt easy to inspect and ensures the expanded
   `%model:#codex` directive is isolated from the VCS/chop/split-file prompt body.

2. Update `tests/test_xprompt_pylimit_split.py` to pin the new generated prompt contract. The existing tests already
   execute the embedded Python with mocked pylimit output and parse the resulting multi-prompt, so they are the right
   place to assert that every generated split-agent segment contains `#m_codex`.

   Add the assertion in the two-file test's existing per-segment loop, and reuse or add a small helper if that keeps the
   deduplication and same-stem collision tests from drifting away from the new contract.

3. Keep existing prompt invariants intact:
   - first segment has no bare `%wait`;
   - later segments remain `%wait`-prefixed;
   - names remain stable and collision-safe;
   - every segment still contains `#gh:sase`, `%group:chop`, `%approve`, and `#split_file:<path>`.

## Validation

1. Run the focused regression suite:

   ```bash
   uv run pytest tests/test_xprompt_pylimit_split.py
   ```

2. Because this repo requires the full check after file changes, run:

   ```bash
   just install
   just check
   ```

3. If the focused test or full check exposes an environment-specific issue around `#m_codex` availability, re-check
   with:

   ```bash
   sase xprompt explain m_codex
   sase xprompt explain codex
   ```

   The intended behavior for this project-local workflow depends on those configured xprompts resolving as described
   above.

## Risks and Mitigations

- Risk: `#m_codex` is a config-provided xprompt rather than a package default. This is acceptable for the project-local
  `xprompts/pylimit_split.yml` workflow because the user explicitly requested this xprompt, but the tests should only
  assert prompt structure rather than require every install to define the alias.
- Risk: adding the model xprompt could accidentally alter wait-chain parsing. Mitigate by preserving the existing
  multi-prompt construction and keeping current wait/name assertions.
- Risk: future edits could reintroduce model-agnostic generated segments. Mitigate by asserting `#m_codex` on every
  generated segment in the focused workflow tests.
