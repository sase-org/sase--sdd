# Auto-archive ChangeSpecs whose PRs were closed without merging

## Problem

ChangeSpec `sase_fix_just_tests_18` (PR https://github.com/sase-org/sase/pull/189) is still `STATUS: Draft` in the
active `~/.sase/projects/sase/sase.gp` file, even though its PR was closed (unmerged) on GitHub on 2026-06-29. The
expectation is that the periodic axe/lumberjack machinery notices when a PR associated with an active ChangeSpec is
merged or closed and transitions the spec to `Submitted` or `Archived` respectively.

## Root cause (diagnosed, with evidence)

The periodic pipeline exists and runs, but it can only detect **merged** PRs. "Closed without merge" is structurally
invisible to it:

1. The `checks` lumberjack (interval 300s, chop `cl_submitted_checks` â†’
   `src/sase/axe/check_cycles.py:run_full_check_cycle`) starts a background `cl_submitted` check per eligible spec.
   Evidence it runs: `~/.sase/axe/logs/lumberjack-checks.log` shows
   `* sase_fix_just_tests_18: Started cl_submitted check` as recently as 2026-07-02 07:09:17.
2. The check script is generated by the GitHub workspace plugin (sase-github linked repo,
   `src/sase_github/workspace_plugin.py::ws_generate_submitted_check_script`):

   ```bash
   state=$(gh pr view <num> --json state -q '.state' 2>/dev/null)
   [ "$state" = "MERGED" ]
   ```

   It exits 0 **only** for `MERGED`. A `CLOSED` PR exits 1 â€” indistinguishable from `OPEN` or from a transient `gh`
   failure.

3. The completion handler (`src/sase/ace/scheduler/checks_runner.py::_handle_cl_submitted_completion`) only acts on exit
   code 0 â†’ transitions to `Submitted`. Every nonzero exit is silently ignored.

So the check for PR #189 has been running every ~5 minutes, exiting 1, and doing nothing. No closedâ†’Archived path exists
anywhere in the periodic machinery (the only CLOSED-awareness is in the interactive submit path, `ws_submit`, which
merely reports an error).

Notably, the status state machine already supports the needed transition: the Rust core planner (`sase-core`,
`crates/sase_core/src/status/planner.rs`) handles `new_status == "Archived"` with `validate=false`, returning
`ARCHIVE_ACTION_TO_ARCHIVE` (which moves the spec to `<project>-archive.gp`), exactly as the existing auto-`Submitted`
path uses it. **No sase-core changes are required.**

## Design

Extend the submitted-check protocol so the script can report three outcomes via exit code, and teach the completion
handler to archive on "closed":

| Exit code | Meaning                     | Handler action                      |
| --------- | --------------------------- | ----------------------------------- |
| 0         | PR merged / CL submitted    | transition â†’ `Submitted` (existing) |
| 20        | PR closed without merging   | transition â†’ `Archived` (new)       |
| other     | open / undetermined / error | no action (existing)                |

Exit code 20 is deliberately distinct from 0/1 so that transient failures (`gh` network error, missing binary, empty
state) can never be misread as "closed" â€” they fall into the default branch and remain no-ops.

This mirrors the existing auto-`Submitted` behavior: a plain status transition with `validate=False` (Draft/Ready/Mailed
â†’ Archived is not in `VALID_TRANSITIONS`, same as Draft â†’ Submitted today), which also performs the move to the archive
file and records the STATUS timestamp. We intentionally do **not** reuse the heavyweight interactive
`archive_changespec` flow (`src/sase/ace/archive.py`) â€” that flow abandons the remote change (the PR is already closed
here), saves diffs, and renames the spec; it is designed for user-initiated archiving of an open PR.

Placement note (Rust core boundary): the closedâ†’Archived rule lives next to the existing mergedâ†’Submitted rule in
`checks_runner.py`. The pure transition decision already routes through the Rust planner; this change only adds a new
caller of that existing API.

## Changes

### 1. sase repo â€” protocol contract

- `src/sase/workspace_provider/_hookspec.py`: define the authoritative constant `SUBMITTED_CHECK_EXIT_CODE_CLOSED = 20`
  and add a docstring to `ws_generate_submitted_check_script` documenting the full exit-code contract (0 = submitted, 20
  = closed-unmerged, anything else = no signal), plus the existing constraint that script bodies must not call `exit`
  (the wrapper in `checks_runner.py::_start_background_check` captures the final `$?`).
- Re-export the constant from `src/sase/workspace_provider/__init__.py`.

### 2. sase repo â€” completion handler

`src/sase/ace/scheduler/checks_runner.py::_handle_cl_submitted_completion`:

- Add a branch: when `exit_code == SUBMITTED_CHECK_EXIT_CODE_CLOSED` and the spec's base status (via `get_base_status`)
  is **not** terminal (not in `sase.status_state_machine.ARCHIVE_STATUSES`), call
  `transition_changespec_status(..., "Archived", validate=False)`, clear the sync-cache entry on success, and return
  `f"Status changed {old_status} -> Archived"` (surfaces in lumberjack logs the same way Submitted transitions do).
- The terminal-status guard is load-bearing: the poll also processes specs living in archive files (e.g. `Reverted`
  specs that may retain a closed PR URL). Without it, a Reverted spec would be silently rewritten to Archived. The
  existing mergedâ†’Submitted branch keeps its current guard (`status != "Submitted"`) â€” tightening it is out of scope.
- Do not gate the new branch on `is_parent_submitted` â€” a closed PR is closed regardless of parent state (checks are
  only _started_ for parent-submitted leaves anyway).

### 3. sase-github linked repo â€” check script

`src/sase_github/workspace_plugin.py::ws_generate_submitted_check_script`: generate a script that distinguishes the
three outcomes without using `exit`:

```bash
state=$(gh pr view <num> --json state -q '.state' 2>/dev/null)
echo "PR state: ${state:-<unavailable>}"
case "$state" in
  MERGED) true ;;
  CLOSED) (exit 20) ;;
  *) false ;;
esac
```

- The `echo` line lands in the check output file (content before the completion marker is ignored by the cl_submitted
  handler) and makes stuck checks diagnosable.
- Hardcode the literal `20` in the generated script with a comment referencing `SUBMITTED_CHECK_EXIT_CODE_CLOSED`,
  rather than importing the constant at module scope. This keeps the plugin importable against an older installed `sase`
  during version skew; a test (below) asserts the values stay in sync.

### 4. Tests

- sase repo, `tests/test_checks_runner.py`:
  - exit 20 on a non-terminal spec â†’ `transition_changespec_status` called with `"Archived"`, `validate=False`; cache
    entry cleared; update message returned.
  - exit 20 on `Reverted`/`Submitted`/`Archived` base statuses (including a workspace-suffixed status) â†’ no transition.
  - exit 1 / other nonzero â†’ no-op (regression guard).
  - exit 0 â†’ `Submitted` (existing behavior unchanged).
  - End-to-end through `process_pending_checks_for` with a fabricated check output file containing
    `===CHECK_COMPLETE=== EXIT_CODE: 20`.
- sase-github repo, `tests/test_workspace_plugin.py`:
  - Execute the generated script in bash with a stubbed `gh` on `PATH` returning `MERGED` / `CLOSED` / `OPEN` / failing
    entirely â†’ assert exit codes 0 / 20 / 1 / 1.
  - Assert the script does not contain a bare `exit` statement (wrapper contract).
  - Assert the literal in the generated script matches `sase.workspace_provider.SUBMITTED_CHECK_EXIT_CODE_CLOSED`.

## Edge cases considered

- **Transient `gh` failures** (network, auth, missing repo/remote, check run with `cwd=None`): `$state` is empty â†’
  default branch â†’ exit 1 â†’ no action. Can never archive spuriously.
- **PR closed then reopened**: if the check fires in the closed window the spec is archived; `sase restore` recovers it.
  Accepted trade-off, same class as manual archive.
- **Running agents/processes on the spec**: the auto-`Submitted` transition already moves specs without killing RUNNING
  entries; the new path stays symmetric. No special handling.
- **Other VCS provider plugins**: the hookspec change is documentation-plus-constant only. Plugins that never emit 20
  keep exactly their current semantics (0 = submitted, else no-op). bare-git has no submitted-check script today and is
  unaffected.
- **Uniform agent runtimes**: not applicable â€” no runtime-specific logic involved.
