# Plan: `sase init memory`

## Goal

Move sibling-repository guidance out of every agent prompt and into generated short-term memory. Add `sase init memory`
so both a project checkout and the user's home agent context get deterministic memory files, required sibling
descriptions, provider instruction shims, and reference validation.

## Phase 1: Config Contract And CLI Skeleton

Owner: one agent in the primary `sase` repo.

Add an `init memory` subcommand beside `init skills`. This phase should:

- Extend the config schema so every `sibling_repos[]` entry requires `description`.
- Update the local `sase.yml` examples/config entries used by this repo so `core`, `github`, `telegram`, and `nvim` have
  useful descriptions.
- Add parser and entry dispatch for `sase init memory`, with a dedicated handler module.
- Reuse the existing config constants for `CONFIG_DIR`, `CHEZMOI_HOME`, and `get_use_chezmoi`.
- Add focused parser/handler smoke tests for the new subcommand.

No memory files should be generated by this phase yet except in tests.

## Phase 2: Memory Generation And Reference Validation

Owner: one agent in the primary `sase` repo.

Implement the handler behavior:

- Generate `memory/short/sase.md` for the current project from only `./sase.yml` sibling entries.
- Generate the home `memory/short/sase.md` from only the global config file (`~/.config/sase/sase.yml`, or the chezmoi
  source file when `use_chezmoi: true`).
- Include a sibling-repos section listing sibling names and descriptions, not paths.
- Include the mandatory instruction that agents must run `sase workspace open -p <sibling_repo> <workspace_num>` and use
  the printed path when a sibling repo needs changes.
- Ensure `memory/long/` exists and write `memory/README.md`.
- Ensure `AGENTS.md` exists. When creating a previously missing `AGENTS.md`, seed a minimal reference to
  `@memory/short/sase.md` so fresh home initialization can pass validation.
- Ensure provider instruction shims exist and contain exactly `@AGENTS.md` for supported provider-specific files such as
  `CLAUDE.md`, `GEMINI.md`, `QWEN.md`, and `OPENCODE.md`.
- Validate that every `memory/short/**/*.md` and `memory/long/**/*.md` file is reachable from `AGENTS.md`, directly or
  through a referenced memory file. Perform this validation after writes and return a non-zero exit code with a concise
  list of unreferenced files.

Add unit tests around local-vs-global sibling selection, missing descriptions, provider shim creation, and transitive
reference validation.

## Phase 3: Workspace Open Semantics And Prompt Cleanup

Owner: one agent in the primary `sase` repo.

Make the generated instruction true and remove the old prompt pollution:

- Change `sase workspace open -p <project> <workspace_num>` so the default behavior materializes, cleans/stashes, checks
  out the default parent, syncs, and then prints the path. Keep `sase workspace path` as the cheap path-only command.
- Update `--print`/`--clean` help and tests so the behavior is explicit.
- Stop appending sibling workspace paths to every launched agent prompt. Keep any env/metadata paths only if they are
  still needed for non-agent-facing bookkeeping such as commit finalization, and avoid exposing them as instructions.
- Update tests that currently assert the prompt note is appended.

## Phase 4: Repo Memory Cleanup And Config Migration

Owner: one agent in the primary `sase` repo, plus the configured chezmoi sibling when global config is edited.

Bring this repo's checked-in memory/config into the new model:

- Remove `memory/long/external_repos.md`.
- Remove the `external_repos` Tier 3 reference and stale dynamic-memory example from `AGENTS.md`.
- Add or preserve references for every remaining memory file, including `memory/long/tui_jk_baseline.md` and nested
  files such as `memory/long/llm_provider_hooks/codex.md` through a referenced parent memory.
- Add descriptions to the global `sibling_repos` entry in the chezmoi-managed config because `use_chezmoi: true` is set.
- Ensure the generated home memory files land under the chezmoi home source tree and are committed/applied by the
  command.

## Phase 5: End-To-End Verification

Owner: final integrating agent.

Run the new command and verification:

- Run `just install` first if dependencies may be stale in this workspace.
- Run targeted tests for init memory, workspace open, sibling prompt cleanup, config schema, and sibling repo behavior.
- Run `sase init memory` from this repo and verify it succeeds.
- Inspect generated `memory/short/sase.md`, `memory/README.md`, provider shims, and home/chezmoi outputs.
- Run `just check` before finishing, unless the only remaining changes are generated markdown/doc files.

## Risks

- `use_chezmoi: true` means the command must write to the chezmoi source tree, commit those changes, and run
  `chezmoi apply --force`; failures here should be surfaced clearly.
- Reference validation must not falsely pass because a file mentions itself.
- `workspace open` behavior changes user-facing semantics, so tests should preserve `workspace path` for path-only use.
- Removing prompt sibling paths may reveal hidden callers that relied on the prompt text rather than config/env; tests
  around agent launch and deferred workspace claims should be updated intentionally.
