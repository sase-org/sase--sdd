# Plan: Preserve Agent Responses in Completion Transcript PDFs

## Problem

The Telegram screenshot shows the completion transcript PDF for `sase-ace_run-260509_004903` with a populated prompt and
an empty `Response` section. The underlying artifacts for that exact run confirm the agent response was not lost:

- `/home/bryan/.sase/projects/sase/artifacts/ace-run/20260509004903/live_reply.md` contains the full agent response.
- `/home/bryan/.sase/projects/sase/artifacts/ace-run/20260509004903/prompt_step_run_docs.json` contains the same
  response under `output._raw`.
- `done.json` points `response_path` at `~/.sase/chats/202605/sase-ace_run-260509_004903.md`.
- That chat file has an empty `## Response` section, which is what the rendered PDF attachment displayed.

The likely root cause is in `src/sase/xprompt/workflow_runner.py`: after a workflow succeeds, `execute_workflow()` sets
`WorkflowResult.response_text` only from the final workflow step when that step output contains `_raw`. The
`refresh_docs` workflow ends with a non-agent `update_marker` step, so the final step has no `_raw` even though the
earlier `run_docs` agent step has the real response. `src/sase/axe/run_agent_exec.py` then saves
`result.response_text or ""` into the ace-run chat transcript, producing the empty PDF.

## Goals

- Completion transcript Markdown/PDF for ace-run workflows should include the most recent agent response, even when
  later non-agent post-steps run after the agent.
- Keep non-agent final-step output behavior intact for `WorkflowResult.output`.
- Avoid changing Telegram-specific code; the bridge correctly attaches the chat transcript PDF it was given.
- Add regression coverage around workflows whose final step has no `_raw` but an earlier agent step does.

## Implementation Approach

1. Update `execute_workflow()` response extraction in `src/sase/xprompt/workflow_runner.py`.
   - Continue deriving `output_str` from the final step exactly as today.
   - Derive `response_text` by scanning completed workflow steps backward for the newest dict output containing a
     non-empty string `_raw`.
   - This preserves existing single-step behavior while covering workflows with post-agent shell/python/update steps.

2. Add focused tests for the workflow result contract.
   - Use existing workflow execution test patterns rather than a full Telegram path test.
   - Cover a workflow where an agent/prompt step emits `_raw` and a later non-agent step emits ordinary structured
     output.
   - Assert `WorkflowResult.response_text` is the earlier `_raw`, while `WorkflowResult.output` remains the final step
     JSON.

3. Add an ace-run finalization regression if practical.
   - Mock `execute_workflow()` to return a `response_text` populated from the earlier agent step and assert
     `save_chat_history()` receives it.
   - If the workflow-level test already exercises the contract cleanly, avoid a brittle runner integration test.

4. Verify.
   - Run targeted tests for workflow execution and ace-run execution.
   - Because this repo requires it after code changes, run `just install` if needed, then `just check` before final
     response.

## Risks and Tradeoffs

- Selecting the newest `_raw` from any completed step may include an embedded workflow or auxiliary prompt step in
  workflows with multiple prompt steps. That matches the user-facing expectation better than requiring the final step to
  be an agent; the completion transcript should represent the latest assistant response generated by the workflow.
- `WorkflowResult.output` should not be changed, because callers may depend on the final structured output from
  post-processing steps.
