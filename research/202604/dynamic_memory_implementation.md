# Dynamic Memory (`memory/dynamic.md`) Implementation Research

## Problem Statement

AGENTS.md defines a three-tier memory system:

- **Tier 1 (always loaded):** `@memory/short/*.md` -- loaded via `@`-import on every session.
- **Tier 2 (dynamic):** `@memory/dynamic.md` -- described as "dynamically generated by sase based on the user's prompt."
  **Not yet implemented.**
- **Tier 3 (long-term):** `memory/long/*.md` -- detailed reference material, listed with descriptions, agent reads on
  demand.

The goal: before an agent session starts, analyze the user's prompt and write a `memory/dynamic.md` file containing only
the tier 3 content relevant to the task. Today, no code does this.

## Architecture Context

### How memory files reach the agent

```
User prompt --> sase TUI/CLI
  --> spawn_agent_subprocess() [launcher.py]
    --> run_agent_runner.py (subprocess, cwd=workspace_dir)
      --> resolve_xprompt_aliases()
      --> process_xprompt_references()    <-- #name expansion
      --> extract_directives_and_write_meta()
      --> prepare_workspace()             <-- workspace is clean
      --> run_execution_loop()
        --> invoke_agent()
          --> ClaudeCodeProvider.invoke()  <-- spawns `claude` CLI
            --> Claude Code reads CLAUDE.md from cwd
              --> @AGENTS.md
                --> @memory/dynamic.md (if it exists on disk)
```

Key constraint: `@memory/dynamic.md` is resolved by the agent runtime (Claude Code, Gemini CLI, etc.) when it reads
AGENTS.md. The file must exist **on disk in the workspace** before the runtime starts.

### The xprompt system

XPrompts are template-based prompt fragments that can be triggered explicitly (`#name` syntax) or discovered
programmatically via tags. Key properties:

- **Loading**: Discovered from 8+ priority-ordered sources (CWD, home, config, plugins, built-in).
- **Fields**: `name`, `content`, `inputs`, `tags`, `description`, `snippet`, `skill`.
- **Tags**: Enum-based semantic roles (`XPromptTag`) used for filtering/routing via `get_by_tag()`.
- **Format**: `.md` files with YAML front matter, `.yml` workflow files, or config entries.
- **Extensible**: Plugins contribute xprompts via entry points; users override via local directories.

The xprompt system is the natural home for dynamic memory because:

1. It already has a content registry with multi-source loading and priority ordering.
2. Tags provide a filtering mechanism that doesn't require explicit `#name` invocation.
3. The content templating (Jinja2) is available if needed.
4. Plugins and users can contribute entries through existing channels.

### Available hook point

After `process_xprompt_references()` expands `#name` refs and before `run_execution_loop()` starts the agent. At this
point we have the fully expanded prompt and a prepared workspace. This is the right place to scan for memory-tagged
xprompts, match them against the prompt, and write `memory/dynamic.md`.

### `@` reference resolution chain

The `@file` import syntax is a feature of the agent runtime (Claude Code, Gemini CLI), not sase. When the runtime reads
CLAUDE.md, it recursively resolves `@` references:

```
CLAUDE.md
  └── @AGENTS.md                          (level 1: runtime resolves)
        ├── @memory/short/build_and_run.md  (level 2: runtime resolves)
        ├── @memory/short/glossary.md
        ├── ...
        └── @memory/dynamic.md              (level 2: runtime resolves, if file exists)
              ├── @memory/long/external_repos.md    (level 3: runtime resolves)
              └── @memory/long/generated_skills.md  (level 3: runtime resolves)
```

The dynamic memory design leverages this: sase writes `@memory/long/foo.md` references into `memory/dynamic.md`, and the
runtime resolves them to actual file content. This is the same mechanism already used for tier 1 memory files at level
2, extended one level deeper for tier 3 content.

### Current tier 3 files (the content pool)

| File                              | ~Lines | Relevant when...                              |
| --------------------------------- | ------ | --------------------------------------------- |
| `memory/long/external_repos.md`   | 25     | Cross-repo work, chezmoi, plugins             |
| `memory/long/generated_skills.md` | 35     | Modifying skill source files, commit workflow |

With only 2 files totaling 60 lines, the current pool is small. The design must accommodate growth.

## Design Constraint: xprompt-based

The implementation should extend the xprompt system with new fields rather than creating a separate registry module.
Memory content is defined as xprompts (with new fields for matching), and the generation step uses existing xprompt
loading infrastructure to discover and filter them.

## Candidate Approaches

### Approach A: `keywords` field + `memory` tag

Add a `keywords: list[str]` field to the `XPrompt` model and a `memory` value to the `XPromptTag` enum. An xprompt
tagged `memory` with keywords becomes a candidate for dynamic inclusion. Before agent launch, all memory-tagged xprompts
are loaded via `get_by_tag(XPromptTag.memory)`, their keywords are matched against the expanded prompt, and matching
content is merged into `memory/dynamic.md`.

**Defining memory xprompts** -- as config entries in `default_config.yml` (or user's `sase.yml`):

```yaml
xprompts:
  memory/external_repos:
    tags: memory
    keywords: [chezmoi, plugin, sase-github, retired Mercurial plugin, sase-telegram, sase-nvim, dotfile, cross-repo]
    content: "@memory/long/external_repos.md"
  memory/generated_skills:
    tags: memory
    keywords: [skill, SKILL.md, init-skills, sase_commit, sase_git_commit, sase_hg_commit, commit workflow]
    content: "@memory/long/generated_skills.md"
```

The key design: `content` uses an `@` reference to the existing tier 3 file rather than embedding the content directly.
When matched, the literal `@memory/long/...` string is written to `memory/dynamic.md`. The agent runtime (Claude Code,
Gemini CLI, etc.) then resolves these `@` references when reading the file as part of the CLAUDE.md/AGENTS.md inclusion
chain, pulling in the actual file content.

This can also be defined as a `.md` file in any xprompt directory:

```markdown
---
tags: memory
keywords: [chezmoi, plugin, sase-github, retired Mercurial plugin, sase-telegram, sase-nvim, dotfile, cross-repo]
---

@memory/long/external_repos.md
```

**Generation logic** (called from `run_agent_runner.py`):

```python
def generate_dynamic_memory(prompt: str, workspace_dir: str, project: str | None) -> None:
    from sase.xprompt.loader import get_all_prompts

    all_prompts = get_all_prompts(project=project)
    matched_content: list[str] = []

    for wf in all_prompts.values():
        if XPromptTag.memory not in wf.tags:
            continue
        if not wf.keywords:
            continue
        if any(kw.lower() in prompt.lower() for kw in wf.keywords):
            # Content is typically an @reference like "@memory/long/foo.md"
            # The agent runtime resolves it when reading dynamic.md
            matched_content.append(wf.steps[0].prompt_part)

    dynamic_path = os.path.join(workspace_dir, "memory", "dynamic.md")
    if matched_content:
        write_file(dynamic_path, "\n\n".join(matched_content))
    elif os.path.exists(dynamic_path):
        os.unlink(dynamic_path)
```

The written `memory/dynamic.md` file might look like:

```markdown
@memory/long/external_repos.md

@memory/long/generated_skills.md
```

Claude Code resolves these `@` references as part of its CLAUDE.md → AGENTS.md → `@memory/dynamic.md` inclusion chain,
the same mechanism that already resolves `@memory/short/*.md`. This is not sase-specific `@` handling -- it is the
runtime's native file inclusion.

**Changes required:**

1. Add `memory` to `XPromptTag` enum (1 line in `tags.py`).
2. Add `keywords: list[str]` field to `XPrompt` dataclass (1 line in `models.py`).
3. Add `keywords` field to `Workflow` dataclass (for uniformity with `xprompt_to_workflow`).
4. Parse `keywords` in `_load_xprompt_from_file()` and `parse_xprompt_entries()` (few lines each).
5. New function `generate_dynamic_memory()` (~30 lines).
6. Call it from `run_agent_runner.py` after xprompt expansion.
7. Add memory xprompt entries to `default_config.yml` with `@` references to existing `memory/long/` files.

**Strengths:**

- Minimal model changes -- one new field, one new tag value.
- Leverages existing infrastructure: `get_all_prompts()` for discovery, tag system for filtering, multi-source loading
  for extensibility.
- Memory xprompts are just regular xprompts with a tag, so all xprompt features (priority ordering, plugin contribution,
  user override) work automatically.
- Keywords are co-located with the content they describe.
- Clean separation: `keywords` = matching mechanism, `memory` tag = routing destination.
- With `@` references as content, tier 3 files remain the single source of truth -- no content duplication. The tier 3
  listing in AGENTS.md stays useful as the fallback path for when keywords don't match.

**Weaknesses:**

- `keywords` field is specific to memory matching today -- it's a new concept on the XPrompt model that nothing else
  uses.
- Keyword matching is brittle for novel phrasings (same as any keyword approach).
- The `memory` tag creates a second "activation path" for xprompts (auto-include vs. explicit `#name`), which is a new
  concept in the system.
- Relies on the agent runtime resolving nested `@` references inside included files. This works today with Claude Code's
  CLAUDE.md inclusion chain, but must be verified for other runtimes (Gemini CLI, Codex).

### Approach B: `keywords` field only (implicit memory behavior)

Add `keywords: list[str]` to `XPrompt` but skip the tag. Any xprompt with a non-empty `keywords` field is automatically
a memory candidate. The presence of keywords IS the signal.

```markdown
---
keywords: [chezmoi, plugin, sase-github]
---

# External Repos

...
```

**Generation logic:**

```python
for wf in all_prompts.values():
    if not wf.keywords:
        continue
    if any(kw.lower() in prompt.lower() for kw in wf.keywords):
        matched_content.append(wf.steps[0].prompt_part)
```

**Strengths:**

- Even simpler -- one new field, no new tag.
- No ambiguity about "what does the `memory` tag mean?"

**Weaknesses:**

- Implicit behavior -- having keywords on an xprompt silently opts it into auto-inclusion, which could surprise users. A
  user might add keywords for documentation purposes without realizing it triggers memory generation.
- No opt-out without removing keywords entirely. With a tag, you can have keywords for other purposes without triggering
  memory behavior.
- Forecloses using `keywords` for other future purposes (e.g., search indexing, tab completion hints).

### Approach C: Dedicated `memory` field with embedded match config

Add a `memory` field (dict or bool) to `XPrompt` that bundles the match configuration. No new tag needed -- the field
itself signals memory behavior.

```markdown
---
memory:
  keywords: [chezmoi, plugin, sase-github]
  priority: 10
---

# External Repos

...
```

Or in config:

```yaml
xprompts:
  memory/external_repos:
    memory:
      keywords: [chezmoi, plugin, sase-github]
      priority: 10
    content: |
      # External Repos
      ...
```

**Generation logic:**

```python
for wf in all_prompts.values():
    if not wf.memory:
        continue
    if any(kw.lower() in prompt.lower() for kw in wf.memory.keywords):
        matched_content.append((wf.memory.priority, wf.steps[0].prompt_part))

matched_content.sort(key=lambda x: x[0], reverse=True)
```

**Strengths:**

- Self-contained -- all memory config is in one field, no tag system involvement.
- Extensible within the field -- can add `priority`, `exclusive` (only include one), `regex` patterns, etc. without
  touching other parts of the model.
- Explicit opt-in -- `memory:` field clearly signals intent.

**Weaknesses:**

- More complex model change -- a nested dict/dataclass field is heavier than a flat `keywords` list.
- Doesn't leverage the tag system, so `get_by_tag()` can't be used for discovery.
- The `priority` and extensibility are speculative -- YAGNI until proven otherwise.
- Parsing nested config adds complexity to `_load_xprompt_from_file()` and `parse_xprompt_entries()`.

### Approach D: Separate xprompt directory convention (`xprompts/memory/`)

No model changes. Instead, use a directory convention: xprompts in a `memory/` subdirectory are memory candidates. Add
`keywords` to front matter for matching.

```
src/sase/xprompts/memory/external_repos.md
src/sase/xprompts/memory/generated_skills.md
```

The loader already namespaces by directory, so these would be `memory/external_repos`, `memory/generated_skills`. The
generation step scans all xprompts whose name starts with `memory/`.

**Strengths:**

- Zero model changes -- relies entirely on naming convention and existing front matter.
- Clear organizational structure -- memory xprompts live in their own directory.

**Weaknesses:**

- Fragile convention -- name-based filtering is less robust than tag-based filtering.
- Doesn't work for config-defined xprompts (no directory structure).
- Users can't contribute memory xprompts from `~/.xprompts/` without matching the naming convention.
- `keywords` field still needs to be added to the model for matching (so "zero model changes" only applies to routing,
  not matching).

## Comparison Matrix

| Criterion                    | A: keywords + tag  | B: keywords only | C: memory field   | D: directory convention |
| ---------------------------- | ------------------ | ---------------- | ----------------- | ----------------------- |
| Model changes                | 2 (field + tag)    | 1 (field)        | 1 (complex field) | 1 (field for keywords)  |
| Explicit opt-in              | Yes (tag)          | No (implicit)    | Yes (field)       | Partial (convention)    |
| Leverages tag system         | Yes                | No               | No                | No                      |
| Works from all sources       | Yes                | Yes              | Yes               | No (needs directory)    |
| Plugin/user extensible       | Yes                | Yes              | Yes               | Partially               |
| Future `keywords` reuse      | Clean (tag scopes) | Blocked          | Clean (scoped)    | Blocked                 |
| Parsing complexity           | Low                | Low              | Medium            | Low                     |
| Discovery via get_by_tag()   | Yes                | No               | No                | No                      |
| Supports `@` ref content [1] | Yes                | Yes              | Yes               | Yes                     |

[1] All approaches can use `@memory/long/foo.md` references as content instead of embedding tier 3 content directly.
This is orthogonal to the routing/matching mechanism. However, Approach A pairs best with this pattern because the
`memory` tag makes the routing intent explicit, preventing accidental auto-inclusion of `@` references meant for other
purposes.

## Recommendation: Approach A (`keywords` field + `memory` tag)

### Justification

1. **Follows existing patterns.** The tag system already serves exactly this purpose -- semantic role routing. Tags like
   `commit`, `propose`, and `vcs` tell the system "this xprompt plays role X." Adding `memory` says "this xprompt's
   content should be auto-included when relevant." This is the same pattern, not a new concept.

2. **Clean separation of concerns.** `keywords` is the matching mechanism. The `memory` tag is the routing destination.
   These are independent: you could later add keywords to non-memory xprompts for search/indexing without triggering
   memory behavior. And you could later add memory xprompts that match on something other than keywords (e.g., LLM
   classification) by keeping the tag but changing the matching strategy.

3. **Leverages existing infrastructure.** Discovery via `get_all_prompts()` + tag filtering gives us multi-source
   loading, priority ordering, and plugin extensibility for free. No new loading code needed.

4. **Minimal and concrete.** Two small additions to the model (`keywords` field, `memory` tag), a parsing tweak in two
   places, and one new ~30-line generation function. No speculative abstractions like priority ordering or regex
   patterns -- those can be added later if needed.

5. **Upgrade path to LLM matching.** When the tier 3 library grows beyond what keywords can handle, the `description`
   field (already on `XPrompt`) provides the semantic summary an LLM classifier needs. Swap the matching function from
   keyword search to LLM call -- the tag-based discovery and file assembly stay the same.

6. **`@` references preserve dual-access to tier 3 content.** By using `@memory/long/foo.md` as the xprompt content
   (rather than embedding the content or moving files), the tier 3 files remain in `memory/long/` where they are already
   listed in AGENTS.md. This gives two access paths: (a) automatic inclusion via dynamic memory when keywords match, and
   (b) on-demand agent reading from the AGENTS.md tier 3 listing when keywords don't match but the agent recognizes the
   task is relevant. No content duplication, single source of truth.

### Proposed changes

1. `src/sase/xprompt/tags.py` -- Add `memory = "memory"` to `XPromptTag` enum.
2. `src/sase/xprompt/models.py` -- Add `keywords: list[str] = field(default_factory=list)` to `XPrompt`.
3. `src/sase/xprompt/workflow_models.py` -- Add `keywords: list[str]` to `Workflow` (propagated via
   `xprompt_to_workflow`).
4. `src/sase/xprompt/loader.py` -- Parse `keywords` from front matter in `_load_xprompt_from_file()`.
5. `src/sase/xprompt/loader_parsing.py` -- Parse `keywords` from config in `parse_xprompt_entries()`.
6. `src/sase/memory/dynamic.py` (new) -- `generate_dynamic_memory(prompt, workspace_dir, project)` function.
7. `src/sase/axe/run_agent_runner.py` -- Call `generate_dynamic_memory()` after xprompt expansion, before execution
   loop.
8. `src/sase/default_config.yml` -- Add memory xprompt entries under `xprompts:` with `tags: memory`, `keywords`, and
   `content: "@memory/long/..."` references to the existing tier 3 files.

### Resolved questions

- **Where should memory xprompts live?** Answer: option (c) -- define xprompts in `default_config.yml` (under
  `xprompts:`) with `content: "@memory/long/foo.md"` references. The `memory/long/` files stay where they are as the
  single source of truth. AGENTS.md keeps its tier 3 listing so agents can still discover and read the files on demand
  when keywords don't match. This avoids duplication, avoids polluting the memory files with front matter, and gives
  dual-access: automatic via dynamic memory + manual via AGENTS.md tier 3 listing. Users can override or extend by
  adding their own memory-tagged xprompts in `sase.yml` or xprompt directories.

### Open questions

- **Should keyword matching be case-insensitive substring or whole-word?** Substring (`"skill" in prompt.lower()`) is
  simpler but risks false positives (e.g., "skill" matching "unskilled"). Whole-word matching with word boundaries
  (`re.search(r'\bskill\b', prompt, re.I)`) is slightly more precise. Start with case-insensitive substring -- the risk
  of over-inclusion is low (60 lines of extra context is cheap) and under-inclusion is the worse failure mode.

- **Should `memory/dynamic.md` be cleaned up after the agent run?** Not necessary -- workspaces are ephemeral clones
  that get cleaned before each run via `prepare_workspace()`. But adding the file to `.gitignore` is prudent to prevent
  accidental commits.

- **Do all supported runtimes resolve nested `@` references?** The CLAUDE.md → AGENTS.md → `@memory/dynamic.md` chain
  already works because Claude Code resolves `@` imports recursively. If `memory/dynamic.md` contains
  `@memory/long/foo.md`, Claude Code should resolve it as a second level of nesting (same as `@memory/short/*.md` in
  AGENTS.md). This needs to be verified for Gemini CLI and Codex. If a runtime doesn't support nested `@` references,
  the generation function could fall back to inlining the file content directly.
